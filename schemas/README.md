# Data Schemas Documentation

This directory contains standardized JSON schemas that define data contracts for the P3 research infrastructure.

## Overview

These schemas ensure consistent data formats across all teams (P1-P5) and enable:
- **Type safety** through validation
- **API contracts** that remain stable over time
- **Documentation** of expected data structures
- **Automated validation** in data pipelines

## Available Schemas

### 1. `molecule_schema.json`
**Purpose**: Standardizes molecular structure data from chemistry workflows.

**Required Fields**:
- `molecule_id`: Unique identifier (format: `MOL-[A-Z0-9]{8,16}`)
- `smiles`: SMILES representation of the molecule
- `source_dataset`: Origin dataset (zinc, chembl, pdbbind, drugbank, etc.)
- `created_at`: ISO 8601 timestamp

**Optional Fields**:
- `atom_features`: Array of atom-level features (element, charge, hybridization, etc.)
- `bond_features`: Array of bond-level features (bond type, conjugation, stereochemistry, etc.)
- `metadata`: Additional properties (molecular weight, LogP, drug-likeness, etc.)

**Use Cases**:
- Ingesting molecules from external databases
- Validating P1 team outputs
- Ensuring consistent molecule representation for ML models

---

### 2. `protein_schema.json`
**Purpose**: Standardizes protein sequence and structure data.

**Required Fields**:
- `protein_id`: Unique identifier (format: `PROT-[A-Z0-9]{8,16}`)
- `sequence`: Amino acid sequence (one-letter code)
- `created_at`: ISO 8601 timestamp

**Optional Fields**:
- `structure_source`: experimental, predicted, homology_model, etc.
- `pdb_reference`: PDB accession code (if available)
- `metadata`: Rich metadata including UniProt ID, organism, domains, secondary structure, mutations, etc.

**Use Cases**:
- Storing protein targets for drug discovery
- Validating structure prediction outputs
- Linking to external databases (PDB, UniProt)

---

### 3. `embedding_schema.json`
**Purpose**: Tracks vector embeddings generated by ML models.

**Required Fields**:
- `embedding_id`: Unique identifier (format: `EMB-[A-Z0-9]{8,16}`)
- `source_id`: ID of source molecule/protein that was embedded
- `model_version`: Version of the embedding model (format: `name@v1.0.0`)
- `vector_dimension`: Dimensionality of the embedding
- `storage_location`: URI/path to the actual vector file
- `created_at`: ISO 8601 timestamp

**Optional Fields**:
- `metadata`: Extensive metadata about model architecture, preprocessing, computation details, validation metrics, etc.

**Use Cases**:
- Tracking embeddings generated by P2 team
- Version control for model outputs
- Enabling reproducibility of ML workflows
- Supporting similarity search and clustering

## Schema Validation

### Python (Pydantic)
Python models are available in `../schemas/models.py`:

```python
from schemas.models import Molecule, Protein, Embedding

# Validate molecule data
molecule = Molecule(
    molecule_id="MOL-ABC12345",
    smiles="CC(=O)OC1=CC=CC=C1C(=O)O",
    source_dataset="chembl",
    created_at="2026-03-02T10:30:00Z"
)

# Access validated data
print(molecule.molecule_id)  # MOL-ABC12345
```

### JSON Validation
Use any JSON Schema validator (v7):

```python
import json
import jsonschema

with open('schemas/molecule_schema.json') as f:
    schema = json.load(f)

# Validate data
jsonschema.validate(instance=data, schema=schema)
```

## Schema Versioning

Schemas follow semantic versioning:
- **MAJOR**: Breaking changes (require data migration)
- **MINOR**: Backward-compatible additions
- **PATCH**: Documentation or clarifications

Current versions:

- `protein_schema.json`: v1.0.0

## Adding New Schemas

When creating new schemas:

1. **Follow JSON Schema Draft 7**: Use `$schema` and `$id` fields
2. **Use descriptive IDs**: Pattern-based IDs with clear prefixes
3. **Document thoroughly**: Include descriptions for all fields
4. **Provide examples**: Add example values
5. **Create Pydantic models**: Update `models.py` with corresponding Python classes
6. **Update validation**: Add validation rules in `../pipelines/validation/`

## Best Practices

### Required vs Optional Fields
- Mark fields as **required** only if they are essential for downstream processing
- Use **default values** for optional fields when appropriate

### Field Naming
- Use `snake_case` for field names
- Be explicit and avoid abbreviations (unless standard in the domain)
- Use consistent naming across schemas

### Metadata Fields
- Keep `metadata` flexible for extensibility
- Document common metadata patterns
- Avoid deeply nested structures (max 2-3 levels)

### ID Formats
- Use prefixed patterns for IDs: `{TYPE}-{ALPHANUMERIC}`
- Ensure IDs are globally unique
- Make IDs human-readable when possible
